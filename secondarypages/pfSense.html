01. Project Overview

A fully segmented network lab built on pfSense running in VirtualBox, divided across three VLANs — LAN (trusted clients), OPT1 (server/DMZ segment), and OPT2 (isolated IoT/lab segment). Each segment has its own DHCP scope, DNS Resolver policy, and independent firewall ruleset enforcing strict inter-VLAN access control.

A site-to-site VPN tunnel simulates a branch-to-HQ scenario, while OpenVPN client access provides secure remote entry to the LAN segment. Suricata runs in inline IDS mode on all interfaces — signatures fire on real traffic during active red team exercises, producing logs that feed into the Mini-SOC Splunk instance.

"A single pfSense box becomes a full enterprise-grade perimeter: segmented VLANs, dual VPN modes, real-time intrusion detection — all running in a home lab."

03. Network Architecture

VLAN segmentation across three interfaces. Traffic between segments is controlled by explicit firewall rules — default deny between OPT1/OPT2, selective allow to LAN services.

04. Firewall Rule Logic

Each interface enforces its own ruleset. LAN has unrestricted outbound. OPT1 permits only specific ports toward LAN services. OPT2 is default-deny — no inter-VLAN traffic unless explicitly allowed.

05. Technical Configuration
STEP 01 — VLAN SEGMENTATION
Three isolated network segments with per-VLAN DHCP and DNS
Each interface (LAN, OPT1, OPT2) operates as an isolated L3 segment. pfSense runs DHCP Server and DNS Resolver independently per interface — a device on OPT2 cannot reach LAN unless an explicit firewall rule permits it.
STEP 02 — FIREWALL RULES
Default-deny between segments, explicit allow for required services
LAN has unrestricted outbound access. OPT1 permits only specific ports toward LAN (e.g., TCP 443 to the Splunk indexer). OPT2 is fully isolated — no inter-VLAN traffic unless a specific rule is added. Floating rules apply Suricata IDS inspection on all interfaces simultaneously.
STEP 03 — DNS RESOLVER + NAT
Per-segment DNS policies and outbound NAT
DNS Resolver (Unbound) is configured with domain overrides per VLAN. Outbound NAT translates each VLAN's subnet to the WAN IP for internet access. DNS rebind protection is active on all segments to prevent internal resource hijacking.
STEP 04 — VPN CONFIGURATION
Site-to-site tunnel + OpenVPN remote access
A site-to-site IPSec VPN simulates a remote branch connecting to the primary LAN. Separately, an OpenVPN server on UDP/1194 handles remote client access — connecting clients land on the LAN segment with full internal routing. Certificate-based authentication only; no shared-secret mode.
STEP 05 — SURICATA INLINE IDS MODE
Active intrusion detection across all VLAN interfaces
Suricata is configured in inline (NFQUEUE) mode rather than passive sniffing — packets are held until Suricata makes a pass/drop decision. ET Open and emerging-threats rulesets are applied. During red team exercises, simulated attacks (nmap scans, Metasploit payloads, brute-force attempts) generated alert events that appeared in Splunk alongside pfSense firewall logs, enabling full kill-chain correlation.

07. Key Takeaways
  • VLAN segmentation limits lateral movement — a compromised OPT2 device cannot reach LAN without an explicit firewall rule, which doesn't exist by default
  • Inline IDS mode provides active blocking capability, not just alerting — Suricata can drop malicious packets in real time rather than logging after the fact
  • Correlating pfSense firewall logs with Suricata events in Splunk gives a full kill-chain view: denied connection attempts alongside matched signatures
  • DNS Resolver per-segment control prevents internal rebind attacks and allows VLAN-specific domain overrides without touching the WAN DNS settings
  • Certificate-based VPN auth eliminates the shared-secret weakness — stolen credentials don't grant VPN access without the corresponding certificate
