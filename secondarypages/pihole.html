01. Project Overview

A network-level DNS sinkhole deployed on a Raspberry Pi Zero 2 WH that intercepts DNS queries across the entire home network, blocking ads, trackers, and known malicious domains before they resolve. Every device — phones, laptops, smart TVs, IoT — is protected automatically with no per-device configuration required.

Beyond ad-blocking, this project feeds DNS query logs directly into the Mini-SOC Splunk pipeline — when a device on the network queries a flagged domain, the event appears in the same dashboard as Suricata IDS alerts, enabling correlation between DNS behaviour and network intrusion data.

03. Installation Process
STEP 01 — System Preparation
Initial setup & network discovery
Scan the local subnet to find the Pi's assigned DHCP address, then SSH in to begin configuration.
sudo nmap -sn 10.0.0.0/24
ssh pi@10.0.0.153
STEP 02 — System Update & Tools
Harden and update before installing anything
Change the default password, pull all OS updates, and install essential utilities.
passwd
sudo apt update && sudo apt full-upgrade -y
sudo apt autoremove -y
sudo apt install -y curl git htop vim net-tools
STEP 03 — Static IP via NetworkManager
Pin the Pi's address so router config stays stable
Use nmtui (text UI) to set a static IP, or manually edit dhcpcd.conf:
# dhcpcd method (legacy, still works)
# Add to /etc/dhcpcd.conf:
interface eth0
static ip_address=10.0.0.153/24
static routers=10.0.0.1
static domain_name_servers=1.1.1.1 8.8.8.8

sudo systemctl restart dhcpcd
ip -4 addr show eth0   # verify
STEP 04 — Install Pi-hole
One-line automated installer
The installer walks through interface selection, upstream DNS, blocklists, web UI, and logging. Selections used:
curl -sSL https://install.pi-hole.net | bash
  • Network interface: eth0
  • Upstream DNS: Cloudflare (1.1.1.1)
  • Default blocklist: StevenBlack (recommended)
  • Admin web interface: Yes
  • Query logging: Enabled (Show everything)
STEP 05 — Set Admin Password & Router DNS
Lock the web UI and redirect all network DNS through Pi-hole
Set the web UI password, then point the router's DHCP DNS field to the Pi's IP. All devices on the network automatically use Pi-hole from this point.
pihole -a -p   # set admin password
Router admin panel → DHCP settings → Primary DNS: 10.0.0.153 → Save & reboot router.
STEP 06 — Blocklist Tuning & Whitelist Management
Reduce false positives, add custom lists
Default lists block ~100k domains. Fine-tuning prevents legitimate sites from breaking while keeping malicious domains blocked.
pihole -g              # update blocklists
pihole -w domain.com   # whitelist a false positive
pihole -t              # live DNS query log (tail mode)
STEP 07 — Security Hardening
Fail2Ban + HTTPS for the admin interface
Install Fail2Ban to block brute-force SSH attempts. Optionally enable HTTPS for the web interface.
sudo apt install fail2ban -y
sudo systemctl enable fail2ban
sudo systemctl start fail2ban
STEP 08 — Verify & Test
Confirm DNS resolution is working through Pi-hole
Query a known ad server — it should return 0.0.0.0 (blocked). Legitimate domains should resolve normally.
nslookup google.com 10.0.0.153        # should resolve
nslookup doubleclick.net 10.0.0.153   # should return 0.0.0.0
04. Results
  • Network-wide ad and tracker blocking across all devices — phones, laptops, smart TVs, IoT sensors
  • ~15,000 DNS queries blocked daily from default + custom blocklists
  • Noticeable reduction in page load times (ad scripts no longer fetched)
  • DNS query log feeds into Mini-SOC Splunk index for SIEM correlation
  • Regex filtering catches DGA-pattern (algorithmically generated) domain queries
  • Custom blocklist additions: known C2 domains from AbuseIPDB and OTX feeds
SIEM INTEGRATION — DNS query logs are forwarded via rsyslog to Splunk on the Windows PC. Blocked domain events appear in the same Splunk dashboard as Suricata IDS alerts, enabling detection of devices querying known-bad domains even when the payload is never downloaded.
05. Key Takeaways
  • DNS-level filtering intercepts threats before any network connection is established — earlier in the kill chain than IDS
  • A single Raspberry Pi running Pi-hole adds negligible latency (<2ms) while protecting every device on the network
  • Integrating Pi-hole logs into Splunk transforms a simple ad-blocker into a network visibility tool
  • Regular blocklist updates are essential — threat actors rotate domains frequently; static lists decay
  • Whitelisting requires ongoing attention — aggressive blocklists break legitimate services regularly
06. Future Enhancements
